# GPUStore Bug 管理清单

> 项目：GPUStore 显卡商城（Vue 3 + Express + MySQL）
> 生成日期：2026-06-22
> 状态说明：✅ 已修复 | 🔧 待验证 | ⚠️ 需关注

---

## 一、后端 API 层

| 编号 | 模块 | Bug 描述 | 严重等级 | 状态 | 修复说明 |
|------|------|----------|----------|------|----------|
| B-001 | 库存管理 | 获取库存列表失败：`Incorrect arguments to mysqld_stmt_execute`，原因是 `pageSize` 为空字符串导致 `parseInt("")` 为 `NaN`，传入 SQL LIMIT 参数失败 | 🔴 严重 | ✅ | 添加默认值与类型校验，并将 `db.execute` 改为 `db.query` |
| B-002 | 订单管理 | 创建订单时查询了不存在的 `products.name` 字段，实际字段为 `title` | 🟠 高 | ✅ | 修改为 `SELECT title AS name, price, stock FROM products` |
| B-003 | 订单管理 | 订单取消接口未回滚库存，导致商品永久丢失 | 🟠 高 | ✅ | 在取消订单事务中增加 `products` 和 `product_summary` 库存回滚 |
| B-004 | 订单管理 | 后端状态机 `validTransitions` 校验逻辑错误，使用 `Object.keys()` 判断目标状态，导致 `completed` 被拦截为"无效目标状态" | 🟠 高 | ✅ | 改为 `Object.values(validTransitions).flat()` 收集所有合法目标状态 |
| B-005 | 订单管理 | `pending` 状态只允许流转为 `paid` 或 `cancelled`，与前端支付成功直接变 `shipped` 的业务逻辑冲突 | 🟠 高 | ✅ | 将 `pending` 允许流转状态调整为 `['shipped', 'cancelled']` |
| B-006 | 订单管理 | 数据库 `orders.status` 枚举缺少 `cancelled` 和 `completed` 状态 | 🟠 高 | ✅ | 执行 ALTER TABLE 扩展 ENUM |
| B-007 | 订单管理 | 后端缺少取消订单、修改状态、删除订单、获取单条订单的接口 | 🟡 中 | ✅ | 在 `orders.js` 中补充 `/cancel`、`/:id/status`、DELETE `/:id`、GET `/:id` |
| B-008 | 订单管理 | 订单创建接口未使用数据库真实价格计算总价，存在前端价格篡改风险 | 🟡 中 | ✅ | 创建订单时从数据库读取 `price` 并计算 `total_price` |
| B-009 | 用户数据 | 地址/收藏等功能未按 `user_id` 隔离，存在越权访问风险 | 🟠 高 | ⚠️ | 需逐接口检查 `user_id` 校验 |
| B-010 | 认证安全 | JWT Secret 硬编码在 `.env` 中，若泄露可导致 Token 伪造 | 🟡 中 | ⚠️ | 建议定期轮换密钥，生产环境使用更强的随机字符串 |
| B-011 | 数据库 | `product_summary` 与 `products` 表未使用事务同步更新，高并发下可能出现数据不一致 | 🟡 中 | ⚠️ | 建议将两表更新纳入同一事务 |

## 二、前端业务层

| 编号 | 模块 | Bug 描述 | 严重等级 | 状态 | 修复说明 |
|------|------|----------|----------|------|----------|
| F-001 | 购物车/订单 | `createOrderAPI` 先创建本地假订单（`id=Date.now()`）并清空购物车，再调后端接口；后端失败时前端数据已丢失 | 🔴 严重 | ✅ | 改为先调后端，成功后再更新前端并清空购物车 |
| F-002 | 订单列表 | `RtOrderList.vue` 的 "待付款"、"待收货"、"待评价" tab 为空，无过滤逻辑和操作按钮 | 🟠 高 | ✅ | 添加 `statusMap`、`statusTabs`、`filteredOrders` 及对应操作按钮 |
| F-003 | 订单列表 | `onConfirmReceive` 调用 `updateOrderStatus` 时传入 `'completed'`，跳过 `delivered` 状态 | 🟠 高 | ✅ | 修改为传入 `'delivered'`，保持 `shipped → delivered → completed` 流转 |
| F-004 | 支付成功 | `RtPaySuccess.vue` 缺少 30 秒倒计时自动送达逻辑 | 🟡 中 | ✅ | 添加 `setInterval` 倒计时，结束自动调用 `updateOrderStatus` 到 `delivered` |
| F-005 | 商品评价 | 评价页面提交后直接模拟成功，未真实更新订单状态，导致订单仍停留在"待评价" | 🟠 高 | ✅ | 提交时调用 `orderAPI.updateOrderStatus(id, 'completed')`，成功后跳转 |
| F-006 | 商品评价 | 评价提交后端返回"无效的目标状态"，因状态机校验拦截 `completed` | 🟠 高 | ✅ | 同 B-004，修复状态机目标状态收集逻辑 |
| F-007 | 商品详情 | "立即购买"按钮未创建真实订单，直接跳转结算页导致无订单数据 | 🟠 高 | ✅ | 改为先调用 `createOrderAPI`，成功后携带 `orderId` 跳转 `/pay` |
| F-008 | 购物车 | "提交订单"按钮逻辑同上，未创建后端订单 | 🟠 高 | ✅ | 同步修改为调后端创建订单后再跳转 |
| F-009 | 结算中心 | 结算页未根据 `orderId` 加载订单详情，展示的是购物车数据而非订单数据 | 🟡 中 | ✅ | `mounted` 中判断 `orderId` 存在则加载订单详情 |
| F-010 | 路由 | 缺少 `/review` 评价页面路由配置 | 🟡 中 | ✅ | 在 `router.js` 中添加 `/review` 路由 |
